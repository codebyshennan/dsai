What is Data Engineering?

  • Data engineering is the discipline of designing, building, and maintaining systems that collect, store, and serve data reliably at scale
  • Goal: ensure downstream analysts and models receive clean, timely, traceable data
  • Without it: analysts spend 80 % of their time cleaning data instead of analysing it
  • Core responsibilities: pipeline development, storage design, data quality, orchestration
  • Common stack: Python, SQL, Airflow (orchestration), dbt (transforms), Spark (scale), cloud object stores
  • Sits between raw operational systems and data consumers (analysts, ML models, dashboards)

The Data Engineer's Role

  • Build pipelines that move data from source systems to storage targets on a schedule
  • Design schemas that balance query performance, flexibility, and maintainability
  • Enforce data quality — validate, monitor, and alert on anomalies automatically
  • Maintain lineage — trace every field back to its source for debugging and compliance
  • Collaborate with analysts to understand access patterns and shape the data model
  • Think in terms of reliability, repeatability, and observability — not just correctness

Data Storage Types

Choosing the right store for the job

Data Science & AI | Skills Union × Localized

Relational Databases and NoSQL

  • Relational (RDBMS): PostgreSQL, MySQL — tables with fixed schema, ACID transactions, best for structured operational data
  • Document stores: MongoDB — schema-flexible JSON documents, good for nested or evolving data
  • Key-value stores: Redis, DynamoDB — ultra-fast lookups by a single key, session/cache data
  • Column-family stores: Cassandra, HBase — optimised for wide rows and time-series writes at scale
  • Graph databases: Neo4j — nodes and edges, ideal for network/relationship data (fraud, recommendations)
  • Choose based on access patterns, schema stability, consistency needs, and scale
Data Science & Analytics | Skills Union

Data Warehouses and Data Lakes

  • Data warehouse: structured, schema-on-write, optimised for analytical queries — Snowflake, BigQuery, Redshift
  • Organised in layers: Staging → Core warehouse → Data marts
  • Star schema: one fact table surrounded by dimension tables — standard warehouse design pattern
  • Columnar storage compresses and scans only the columns needed — 10–100× faster for analytics
  • Data lake: raw data in object storage (S3, GCS) — schema-on-read, supports all formats (JSON, Parquet, CSV)
  • Data lakehouse (Delta Lake, Iceberg) blends lake flexibility with warehouse-grade ACID guarantees
  • Rule: warehouse for curated analytics; lake for raw archival and ML feature engineering

ETL Fundamentals

Extract, Transform, Load

Data Science & AI | Skills Union × Localized

Extract: Getting Data from Sources

  • Extract = read data from one or more source systems into the pipeline
  • Relational databases: SQL query via SQLAlchemy / psycopg2; full or incremental by updated_at
  • REST APIs: requests.get(url, headers=...) with pagination and rate-limit handling
  • Files: CSV, JSON, Parquet from local disk, SFTP, or cloud object storage
  • Streaming sources: Kafka topics, webhooks — consumed in near-real-time
  • Key challenges: authentication, pagination, schema drift, network failures, rate limits
  • Always implement retry logic with exponential back-off for API sources
Data Science & Analytics | Skills Union

Transform: Cleaning and Enriching

  • Transform = apply business rules to turn raw data into analysis-ready data
  • Data cleaning: fix types, standardise formats, handle nulls and duplicates
  • Data enrichment: join with reference tables, derive new columns, apply business rules
  • Schema normalisation: flatten nested structures, rename columns to a consistent convention
  • Aggregation: roll up rows to the right grain (e.g. daily sales per product)
  • Keep transforms idempotent — running the same transform twice should produce the same output
  • Test transforms with unit tests before deploying to production pipelines
Data Science & Analytics | Skills Union

Load: Delivering to Targets

  • Load = write transformed data to the target storage system
  • Full load: truncate and reload the entire table — simple but slow for large tables
  • Incremental load: append or upsert only new/changed rows — efficient for large datasets
  • Merge / upsert: update matching rows, insert new ones — preserves history
  • Use transactions to ensure the target is either fully updated or not at all — avoid partial writes
  • Bulk loading (COPY, INSERT SELECT) is orders of magnitude faster than row-by-row inserts
  • ELT alternative: load raw data first, then transform inside the warehouse with SQL or dbt

Data Pipeline Design Patterns

Batch, streaming, idempotency, and lineage

Data Science & AI | Skills Union × Localized

Batch vs Streaming Pipelines

  • Batch: process data in fixed windows (hourly, daily) — simpler, lower cost, higher latency
  • Streaming: process events as they arrive (Kafka, Flink) — low latency, higher complexity
  • Most organisations start with batch; add streaming only when latency requirements demand it
  • Micro-batch (Spark Structured Streaming) is a middle ground: short windows, near-real-time
  • Scheduling: cron expressions (Airflow, cron) trigger batch runs; event triggers drive streaming
  • Design for failure tolerance: pipelines will fail — build retry, alerting, and dead-letter handling in from day one
Data Science & Analytics | Skills Union

Idempotency and Data Lineage

  • Idempotency: running a pipeline run multiple times produces the same result — critical for safe retries
  • Achieve idempotency by using insert-or-replace / merge instead of plain INSERT
  • Partitioned tables: write each run to a dated partition — re-running overwrites only that partition
  • Data lineage: the ability to trace a field from its destination all the way back to its source system
  • Lineage is essential for debugging data quality issues and satisfying compliance audits
  • Tools like dbt auto-generate lineage graphs from SQL model dependencies
  • Document SLAs: when should each dataset be available, and what happens if it is late?

Schemas, APIs, and Change Data Capture

  • Schema management: define and version schemas (Avro, JSON Schema, Protobuf) to detect drift early
  • Schema evolution: add nullable columns freely; removing or renaming columns is a breaking change
  • REST API integration: pagination (cursor, offset), authentication (OAuth2, API keys), rate limits
  • Change Data Capture (CDC): stream only changed rows from a database (Debezium reads the WAL / binlog)
  • CDC is far more efficient than polling — used to sync OLTP databases into data warehouses in near-real-time
  • Event-driven integration: publish domain events to a message bus (Kafka); consumers subscribe independently
  • Prefer idempotent consumers so duplicate events do not corrupt the target

Data Quality in Pipelines

Validate, monitor, alert

Validation, Monitoring, and Alerting

  • Validation rules at ingestion: schema check, null check, range check, referential integrity
  • Run checks before writing to the target — reject or quarantine bad records, never silently drop them
  • Row count checks: compare source count to target count after each load run
  • Statistical monitoring: alert if column mean, null rate, or unique count shifts beyond a threshold
  • Great Expectations: open-source library for defining and running data validation suites
  • Alerting: route failures to Slack / PagerDuty so the on-call team can respond within SLA
  • Track data freshness: last updated timestamp — stale data is as bad as incorrect data

Real-World Tools Overview

What each tool solves

Data Science & AI | Skills Union × Localized

Airflow, dbt, and Spark

  • Apache Airflow: workflow orchestration — define DAGs of tasks in Python, schedule and monitor runs
  • Airflow solves: *how do I run task B only after task A succeeds, on a schedule, with retries?*
  • dbt (data build tool): SQL-first transformation layer inside the warehouse — SELECT in, table/view out
  • dbt solves: *how do I manage, test, and document all my SQL transforms as version-controlled code?*
  • Apache Spark: distributed in-memory compute for datasets too large for a single machine
  • Spark solves: *how do I process terabytes of data across a cluster without writing custom MapReduce?*
  • Most modern stacks combine all three: Airflow schedules, dbt transforms, Spark handles large-scale ETL

Data Engineering Best Practices

  • Modular pipelines: each task does one thing well — easier to test, debug, and reuse
  • Version control everything: pipeline code, SQL transforms, and schema definitions belong in git
  • Test before you deploy: unit test transforms, integration test the full pipeline on a sample
  • Log everything: task start/end, row counts, warnings — logs are your first debugging tool
  • Monitor proactively: dashboards showing pipeline health, SLA status, and quality metrics
  • Automate quality gates: block a load if the row count drops > 10 % versus yesterday
  • Start simple, then scale: a well-structured batch job beats a complex streaming system that no one understands
