Relational Databases: Core Concepts (1/2)

  • Tables, Rows, and Columns:
  • - A table is like a spreadsheet: rows are records, columns are attributes
  • - Every row represents one entity (e.g. one customer, one order)
  • - Every column holds one type of value (e.g. name, date, amount)
  • Keys and Relationships:
  • - Primary key — uniquely identifies each row in a table
  • - Foreign key — a column that references the primary key of another table
  • - Keys enforce referential integrity — no orphaned records
  • Schema:
  • - The blueprint of a database: table names, column names, data types, constraints

The SELECT Statement (1/2)

  • SELECT retrieves data — the most common SQL operation
  • Core clauses (in writing order):
  • - `SELECT` — which columns to return
  • - `FROM` — which table to query
  • - `WHERE` — filter rows (optional)
  • - `ORDER BY` — sort the result (optional)
  • - `LIMIT` — cap the number of rows returned (optional)
  • Examples:
  • - `SELECT * FROM customers;` — all columns, all rows
  • - `SELECT first_name, email FROM customers;` — specific columns

The SELECT Statement (2/2)

  • - `SELECT * FROM customers ORDER BY last_name ASC LIMIT 10;`

Writing and Reading Data (CRUD) (1/2)

  • CREATE TABLE — defines structure with data types and constraints
  • - `SERIAL PRIMARY KEY` auto-increments the ID
  • - `NOT NULL` rejects missing values
  • - `UNIQUE` prevents duplicate values in a column

  • INSERT INTO — adds one or multiple rows

  • - Multi-row: list multiple value tuples in one statement (more efficient)
  • UPDATE — modifies existing rows (always use WHERE!)
  • DELETE — removes rows (always use WHERE!)

WHERE Clause and Logical Operators (1/2)

  • WHERE filters rows before they reach the output
  • Comparison operators: `=`, `!=`, `<`, `>`, `<=`, `>=`
  • Logical operators:
  • - `AND` — both conditions must be true
  • - `OR` — at least one condition must be true
  • - `NOT` — inverts a condition
  • Range and membership:
  • - `BETWEEN 10 AND 50` — inclusive range check
  • - `IN ('A', 'B', 'C')` — match any value in a list
  • - `NOT IN (...)` — exclude a list of values

  • - `LIKE '_ohn'` — _ matches exactly one character
  • - `ILIKE` — case-insensitive version (PostgreSQL)

Handling NULL Values (1/2)

  • NULL means the value is unknown or missing — not zero, not empty string
  • NULL arithmetic always returns NULL: `NULL + 1 = NULL`
  • Checking for NULL:
  • - `WHERE column IS NULL` — find missing values
  • - `WHERE column IS NOT NULL` — find present values
  • - Never use `= NULL` — it always returns false
  • Working with NULL in output:
  • - `COALESCE(column, 0)` — return 0 if column is NULL
  • - `NULLIF(a, b)` — return NULL if a equals b (useful to avoid divide-by-zero)

Aggregate Functions (1/2)

  • Aggregations collapse many rows into a summary value
  • Think of it as: individual receipts → monthly spending report
  • Core aggregate functions:
  • - `COUNT(*)` — count all rows in the group
  • - `COUNT(col)` — count non-NULL values
  • - `COUNT(DISTINCT col)` — count unique non-NULL values
  • - `SUM(col)` — total of a numeric column
  • - `AVG(col)` — mean value (ignores NULLs)
  • - `MIN(col)` / `MAX(col)` — lowest / highest value

Aggregate Functions (2/2)

  • - FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

GROUP BY and HAVING (1/2)

  • GROUP BY partitions rows into groups before aggregation
  • - Every column in SELECT must be either in GROUP BY or wrapped in an aggregate function
  • - Example: `SELECT category, COUNT(*) FROM products GROUP BY category;`
  • - Group by multiple columns: `GROUP BY year, month`
  • HAVING filters groups after aggregation
  • - `WHERE` cannot reference aggregate results — use `HAVING` instead
  • - Example: find categories with more than 100 products:
  • - `GROUP BY category HAVING COUNT(*) > 100`
  • Common patterns:
  • - Top N groups: `GROUP BY … ORDER BY SUM(amount) DESC LIMIT 5`

Why Joins Matter (1/2)

  • Almost every real question spans more than one table
  • Examples: customers + orders, students + enrolments, products + reviews
  • The key decision: which rows are you willing to *drop* from the result?
  • Getting this wrong silently loses data or creates duplicates
  • Join syntax:
  • - `FROM table_a a JOIN table_b b ON a.id = b.a_id`
  • - Use short aliases (`a`, `b`, `c`) to keep queries readable
  • - Always qualify column names with their table alias in multi-table queries
  • Cartesian product warning:
  • - Missing `ON` clause → every row of A matched with every row of B
Data Science & Analytics | Skills Union

Why Joins Matter (2/2)

  • - 1,000 × 1,000 rows = 1,000,000 rows — almost never what you want
Data Science & Analytics | Skills Union

INNER JOIN

  • INNER JOIN returns only rows that have a match in *both* tables
  • Visual: the overlapping centre of two Venn diagram circles
  • Rows with no match on *either* side are dropped from the result
  • Use when: you only care about records that exist in both tables
  • Example:
  • - Orders with their customer name → only orders that have a matching customer
  • - Customers with no orders and orders with no customer are both excluded
  • Syntax:
  • - `FROM orders o INNER JOIN customers c ON o.customer_id = c.customer_id`
  • - `INNER` keyword is optional — plain `JOIN` defaults to INNER
Data Science & Analytics | Skills Union

LEFT JOIN and RIGHT JOIN (1/2)

  • LEFT JOIN — all rows from the *left* table, matched rows from the right
  • - Unmatched right-side columns appear as NULL
  • - Use when: you want all left-side records regardless of whether they have a match
  • - Example: all customers, with order count (0 for customers with no orders)
  • RIGHT JOIN — all rows from the *right* table, matched rows from the left
  • - Mirror image of LEFT JOIN
  • - Tip: rewrite any RIGHT JOIN as a LEFT JOIN by swapping table order — easier to read
  • LEFT JOIN anti-pattern (find missing matches):
  • - `LEFT JOIN … WHERE right_table.id IS NULL`
  • - Returns only left-side rows that had *no* match in the right table

LEFT JOIN and RIGHT JOIN (2/2)

  • - Example: customers who have never placed an order

FULL OUTER JOIN and Join Summary (1/2)

  • FULL OUTER JOIN — all rows from *both* tables
  • - Unmatched columns from either side appear as NULL
  • - Use when: you want a complete picture of both datasets with gaps visible
  • - Example: audit two systems side-by-side to see where they disagree
  • Join type summary:
  • - INNER JOIN → only matching rows from both sides
  • - LEFT JOIN → all left rows + matching right rows (NULL if no match)
  • - RIGHT JOIN → all right rows + matching left rows (NULL if no match)
  • - FULL OUTER JOIN → all rows from both sides (NULL where no match)

Subqueries (1/2)

  • A subquery is a SELECT nested inside another SQL statement
  • In the WHERE clause:
  • - `WHERE customer_id IN (SELECT id FROM vip_customers)`
  • - Returns rows where the column matches the subquery result
  • In the FROM clause (derived table):
  • - `FROM (SELECT … FROM …) AS sub` — treat a query result as a table
  • - Useful for multi-step calculations before a final SELECT
  • Correlated subquery:
  • - References the outer query's current row — runs once per outer row
  • - Powerful but can be slow on large tables
Data Science & Analytics | Skills Union

Subqueries (2/2)

  • EXISTS / NOT EXISTS:
  • - `WHERE EXISTS (SELECT 1 FROM orders WHERE orders.customer_id = c.id)`
  • - Stops scanning as soon as one match is found — often faster than IN

CTEs (Common Table Expressions) (1/2)

  • A CTE (WITH clause) names a subquery so you can reference it like a table
  • Why use CTEs:
  • - Breaks a complex query into readable, named steps
  • - Avoids repeating the same subquery multiple times
  • - Easier to test each step individually
  • - Supported by all major databases (PostgreSQL, MySQL 8+, SQL Server, BigQuery)
  • Syntax:
  • - `WITH step_name AS (SELECT …) SELECT … FROM step_name`
  • - Chain multiple CTEs: `WITH a AS (…), b AS (… FROM a) SELECT … FROM b`

CTEs (Common Table Expressions) (2/2)

  • - Reference themselves to walk hierarchical data (org charts, folder trees)
  • - `WITH RECURSIVE tree AS (base_case UNION ALL recursive_step)`

Window Functions (1/2)

  • Window functions compute values across a set of rows *related to the current row* without collapsing them into one
  • Unlike GROUP BY, original rows are preserved in the output
  • Syntax: `function() OVER (PARTITION BY … ORDER BY …)`
  • Ranking functions:
  • - `ROW_NUMBER()` — unique sequential rank (no ties)
  • - `RANK()` — same rank for ties, gaps after tied ranks
  • - `DENSE_RANK()` — same rank for ties, no gaps
  • Running totals and moving averages:
  • - `SUM(amount) OVER (PARTITION BY customer_id ORDER BY date)` — running total per customer
  • - `AVG(amount) OVER (ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)` — 7-day moving average

Best Practices and Query Optimisation

Write clean, fast, maintainable SQL

Writing Readable SQL (1/2)

  • Formatting conventions:
  • - Capitalise keywords: `SELECT`, `FROM`, `WHERE`, `JOIN`
  • - One clause per line for queries longer than one line
  • - Indent continuation lines consistently
  • Naming and aliases:
  • - Use descriptive aliases: `customers AS c`, not just `x`
  • - Name calculated columns clearly: `SUM(amount) AS total_revenue`
  • - Avoid `SELECT *` in production — list columns explicitly
  • CTEs over nested subqueries:
  • - Replace deeply nested subqueries with named CTEs
Data Science & Analytics | Skills Union

Writing Readable SQL (2/2)

  • - Each CTE should do one logical thing
  • Comments:
  • - `-- single line` and `/* block */` comments explain *why*, not just what
  • - Document non-obvious business logic and edge cases
Data Science & Analytics | Skills Union

Query Optimisation Tips (1/2)

  • Filter early: push WHERE conditions as close to the source table as possible
  • SELECT only what you need: retrieving fewer columns reduces I/O
  • Indexes:
  • - An index lets the database find rows without scanning the whole table
  • - Index columns used in WHERE, JOIN ON, and ORDER BY
  • - Too many indexes slow down INSERT/UPDATE/DELETE
  • Join order: place the more selective (smaller result) table first in large joins
  • EXPLAIN / EXPLAIN ANALYZE:
  • - Shows the query execution plan before (or after) running
  • - Look for `Seq Scan` on large tables — often a missing index
Data Science & Analytics | Skills Union

Query Optimisation Tips (2/2)

  • - `EXPLAIN ANALYZE` runs the query and shows actual vs estimated row counts
  • Avoid functions on indexed columns in WHERE:
  • - `WHERE LOWER(email) = 'x'` cannot use a plain index on `email`
  • - Use a functional index or rewrite the condition
Data Science & Analytics | Skills Union

Data Integrity and Maintenance (1/2)

  • Constraints keep data trustworthy:
  • - `NOT NULL` — no missing required values
  • - `UNIQUE` — no duplicate entries in a column
  • - `FOREIGN KEY … REFERENCES` — no orphaned records
  • - `CHECK` — custom business rules (e.g. `price > 0`)
  • Transactions:
  • - Wrap related changes in `BEGIN … COMMIT` so they succeed or fail together
  • - Use `ROLLBACK` to undo on error
  • Schema changes:
  • - Version-control migration files (e.g. Flyway, Liquibase, Alembic)

Data Integrity and Maintenance (2/2)

  • - Test migrations in a staging environment before production
  • Ongoing maintenance:
  • - Monitor slow queries with `pg_stat_statements`
  • - Regular `ANALYZE` keeps query planner statistics current
  • - Review and drop unused indexes to reduce write overhead
